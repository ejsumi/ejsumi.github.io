# From Prompts to Agents: A Technical Writer's Guide to GitHub Copilot

Most articles explain AI agents as autonomous systems that plan, reason, and act. That framing is technically correct but practically useless if you're a technical writer whose primary AI tool is GitHub Copilot. This post starts where you actually are — with a prompt — and walks through exactly how prompts, instructions, skills, tools, and agents differ, with DITA and Markdown examples throughout.



---

## The Core Question



> If I can already ask GitHub Copilot to rewrite a DITA topic, why do I need an agent?

The answer is scope. A prompt handles one request. An agent works toward a goal. Everything in between — instructions, skills, tools — is what makes that possible.



---



## The Five Building Blocks



| Building Block | What It Is | Technical Writing Equivalent |
|---|---|---|
| **Prompt** | A single, one-time request | A question you ask a colleague |
| **Instructions** | Rules that shape every response | Your team's style guide |
| **Skill** | Packaged expertise for a specific task | A review checklist |
| **Tool** | Ability to interact with files and systems | Your authoring software |
| **Agent** | All of the above, working toward a goal | A team member assigned to a task |



---



## Level 1: Prompt



A prompt is a single input that produces a single output. It has no memory, no context beyond what you type, and no persistent rules.



```text

Rewrite this procedure in the active voice.

```



**Characteristics:**

- One-time request

- No memory between sessions

- No tools or file access

- No persona or role

- No persistent rules


**The limitation:** Every time you start a new session, Copilot forgets everything. You repeat yourself constantly.



---



## Level 2: Instructions

Instructions define how Copilot should behave — persistently, across every prompt in that context.

In GitHub Copilot, instructions live in a file at the root of your repository:

`.github/copilot-instructions.md`


```markdown

You are a technical writer following DITA 1.3 standards.



- Use active voice.

- Keep sentences under 20 words.

- Always generate a <shortdesc> for every topic.

- Use "select" not "click" for UI interactions.


```

Once this file exists, every prompt in that repo follows these rules. You stop repeating yourself.

**Watch out:** Instructions are repo-scoped. If your DITA files and Markdown files live in separate repositories, you need a separate instructions file for each.



---

## Level 3: Skills



A skill packages expertise for a specific, repeatable task. Instead of a generic assistant, Copilot has a defined capability it can apply consistently.



**Example — DITA Review Skill:**



```text

Analyze the open DITA topic and check for:

- Missing <shortdesc>

- Invalid or broken <xref> elements

- Missing or incorrect metadata attributes

- Style guide violations (active voice, sentence length, terminology)



Return a structured list of issues found.

```

**The difference from a prompt:** A skill is reusable. You define it once and invoke it repeatedly across different topics without rewriting the logic each time.



---



## Level 4: Tools

Tools are what allow Copilot to move from talking to acting. Without tools, Copilot can only generate text in response to what you paste in. With tools, it can interact with your actual files and systems.


| Tool | What It Does |
|---|---|
| Read files | Access DITA or Markdown files in your workspace |
| Search workspace | Find content across your entire repository |
| Create files | Generate new topic files |
| Edit files | Make changes directly to existing content |
| Run scripts | Execute validation or transformation scripts |
| Access Git | Read commit history, branches, file changes |


**The key distinction:**

```

Without tools → AI can only talk.

With tools    → AI can act.

```

In GitHub Copilot, the `@workspace` agent gives you file-level access to your repository.

---



## Level 5: Agent



An agent combines all four building blocks and works toward a goal — not just a single response.

**Agent = Persona + Instructions + Skills + Tools + Goal**

### Example: DITA Documentation Reviewer Agent



| Component | Value |
|---|---|
| **Persona** | Senior Documentation Architect |
| **Instructions** | Follow DITA 1.3 standards and team style guide |
| **Skills** | DITA validation, metadata review, link analysis |
| **Tools** | Read files, search repository, create reports |
| **Goal** | Review all DITA topics in this release and generate a findings report |

The agent doesn't wait for you to ask about each topic individually. It works through the goal systematically.

---



## How the Scope Expands: Side-by-Side Examples

| Building Block | DITA Example | Markdown Example |
|---|---|---|
| **Prompt** | `Create a shortdesc for this topic.` | `Write a one-sentence description for this page.` |
| **Instructions** | `Always write shortdesc as one sentence under 50 words.` | `Every file must include frontmatter: title, description, tags, last_updated.` |
| **Skill** | Analyze topic → generate shortdesc + metadata + index terms | Analyze page → generate frontmatter + suggest tags + check heading hierarchy |
| **Agent** | Review entire doc set → identify missing shortdesc, broken xrefs, duplicate topics → generate report | Audit docs folder → find missing frontmatter, broken links, heading violations → generate report |

## What Agents Are Not


| What people assume | What's actually true |
|---|---|
| A better prompt | No — an agent is a different scope entirely, not a smarter single response |
| A chatbot | No — a chatbot responds; an agent acts toward a goal |
| A replacement for writers | No — agents depend on good instructions, quality source content, and human review |
| Fully autonomous | No — agents work within defined boundaries and still require governance |



Agents still depend on:

- Well-written instructions

- Quality source content

- Human review of outputs

- Proper content governance


Garbage in, garbage out applies here more than anywhere else.

---

## Key Takeaways



| Concept | Remember This |
|---|---|
| Prompt | One request, one response, no memory |
| Instructions | Persistent rules — define once, apply everywhere in that repo |
| Skill | Reusable, task-specific expertise |
| Tool | What gives the AI the ability to act on files, not just generate text |
| Agent | All of the above, working toward a goal |
| Instructions are repo-scoped | Multi-repo teams need multiple `.github/copilot-instructions.md` files |
| Agents don't replace governance | They extend it — your style guide, metadata standards, and human review are important |



Most technical writers don't need to build sophisticated agents from scratch. The biggest gains come from writing good instructions, defining reusable skills, and establishing clear documentation workflows. Once those exist, GitHub Copilot agents become a natural extension of what your team already does.
